# Prepare Mobile Release

## Objective

Assess and prepare human-controlled release readiness using reproducible unsigned or simulator-safe evidence, without signing, uploading, submitting, publishing, deploying, distributing, spending money, or using real credentials.

## Trigger

Run only when the user explicitly names prepare-mobile-release or clearly says to invoke this Skill. Never trigger automatically from ordinary feature completion, a version discussion, or a build request. If invocation is not explicit, stop and ask the user to invoke it manually.

## Inputs

- release target, platforms, version, build number, channel, and intended date;
- frozen repository snapshot and complete change summary;
- approved versioning, variant, flavor, scheme, packaging, privacy, compliance, and store-metadata policies;
- successful implementation, test, security, accessibility, performance, and code-review evidence;
- unsigned or simulator-safe build commands discovered from the project;
- named human release owner and signing or publication process that will occur outside this Skill.

Do not request real certificates, provisioning profiles, signing identities, keystores, passwords, tokens, service accounts, or store credentials.

## Supported technologies

Android, iOS, KMP applications through their hosts, Flutter, and React Native. Libraries and packages may receive readiness review only; publication remains prohibited.

## Preconditions

- Manual invocation is explicit.
- Scope is frozen and repository changes are fully understood.
- Required feature and defect work is complete.
- All required reviews and tests have current evidence.
- Version policy, target variants or schemes, and human release owner are known.
- Validation can use unsigned, simulator, or other non-publishing paths without real credentials.

## Primary owner and reviewers

mobile-release-engineer is primary and cannot perform final code review. mobile-code-reviewer must already have reviewed the implementation. mobile-security-reviewer reviews privacy, permissions, dependencies, credentials boundaries, and store declarations. mobile-ui-accessibility-reviewer and mobile-performance-reviewer review applicable evidence. Technology owners explain platform configuration. A human release owner makes every release decision and performs any later signing or external action.

## Ordered workflow

1. Confirm explicit invocation, target platforms, channel, version, build number, scope freeze, human owner, and absolute prohibition on external release actions.
2. Inspect repository instructions, status, changed files, version declarations, variants, flavors, schemes, targets, manifests, entitlements, privacy files, dependencies, lockfiles, and release documentation.
3. Build a release traceability matrix covering requirements, fixes, migrations, flags, permissions, privacy, analytics, localization, accessibility, performance, dependencies, known issues, and rollback.
4. Verify that implementation and mobile-code-reviewer reviews are complete and that the release engineer did not implement the code being assessed.
5. Reconcile version and build identifiers across all affected native hosts and cross-platform configuration without editing unless a specific metadata edit is separately approved.
6. Discover safe build, build-for-testing, assemble, archive-validation, or package commands from repository configuration. Exclude signing, notarization, upload, submission, deployment, publication, and release creation.
7. Run only explicitly approved unsigned or simulator-safe configuration, compilation, analysis, formatting, test, and packaging validation.
8. Inspect outputs locally without transmitting them. Record paths and hashes only when safely generated by existing tools; do not retain or distribute sensitive artifacts.
9. Review warnings, dependencies, permissions, entitlements, privacy declarations, localization, accessibility, performance, offline behavior, migrations, monitoring readiness, and rollback.
10. Produce a blocked or ready-for-human-release report. Ready means evidence is sufficient for the human process, not that a release occurred.
11. Obtain security and relevant specialist confirmation, complete triple review, and perform final verification.
12. Hand off a checklist to the human release owner. Stop before signing credentials, store consoles, uploads, deployment, distribution, publication, or cost.

## Conditional steps

- Android: verify variants, manifest merge, permissions, version fields, dependency locks, lint, tests, and unsigned build outputs; never access a keystore or publishing task.
- iOS: verify project or workspace, shared scheme, version fields, entitlements, privacy manifest, tests, and simulator-safe build or build-for-testing; never access signing identities or upload an archive.
- KMP: verify affected targets, shared and host tests, expect and actual coverage, and native host version alignment.
- Flutter: verify flavors, package and lock consistency, analyze and tests, host configuration, and non-publishing build validation.
- React Native: verify package manager and lockfile, type and lint checks, tests, Metro, host configuration, and safe native host builds.
- Migration: require backup, forward and rollback behavior, recovery evidence, compatibility, and human operations plan without executing production migration.

## Validation gates

- Gate 1: manual invocation, scope, version policy, target, and human owner are explicit.
- Gate 2: all implementation and independent review evidence is current and complete.
- Gate 3: version, variants, schemes, dependencies, privacy, permissions, and metadata are consistent.
- Gate 4: every applicable unsigned or simulator-safe check passes with no unexplained warning.
- Gate 5: security, accessibility, performance, migration, and rollback evidence is complete or blocks readiness.
- Gate 6: final report confirms no signing, credential access, external write, upload, submission, publication, deployment, distribution, cost, or destructive action.

## Failures

Any missing required evidence, warning, unresolved finding, version mismatch, build failure, credential prompt, signing dependency, or external action blocks readiness. Preserve exact evidence and do not change signing, disable checks, waive findings, or switch to a publishing path.

## Stop conditions

Stop immediately for absent manual invocation, dirty or ambiguous scope, unresolved required review, unavailable mandatory check, unknown version policy, credential or signing request, store or production access, external upload, submission, deployment, distribution, publication, release creation, financial action, or destructive command.

## Evidence

Record manual invocation, repository snapshot, change inventory, versions, targets, commands, tool versions, exit statuses, warnings, test and review results, unsigned artifact inventory, criterion ledger, blockers, and human hand-off.

## Outputs

- release-readiness matrix and blocked or ready-for-human-release conclusion;
- version, configuration, dependency, privacy, permission, and metadata findings;
- exact unsigned validation evidence;
- known issues, rollback and monitoring checklist;
- explicit human-only signing and release steps described at a policy level without credentials or execution.

## Acceptance criteria

Readiness may be reported only when all required evidence passes, version and configuration are consistent, risk has named human ownership, and the report clearly states that no release action occurred. An unavailable required check results in blocked, not ready.

## Human approvals

Humans control invocation, scope freeze, versioning, privacy and compliance declarations, risk acceptance, metadata, signing prerequisites, artifact retention, and every later signing, upload, submission, deployment, distribution, publication, or financial action.

## Prohibited actions

Never sign, import credentials, access keystores or signing identities, notarize, upload, submit, publish, deploy, distribute, create a release, send artifacts, spend money, change production, bypass validation, destroy data, or run Git write commands.
