# SDLC Requirements Threat Modeling

1. Confirm product scope, lifecycle stage, owners, reviewer, approver, evidence sources, and decision dates.
2. Map SDLC controls, security requirements, trust boundaries, data flows, assets, user roles, misuse cases, and abuse paths.
3. Identify design obligations, test obligations, release gates, assumptions, gaps, and dependencies.
4. Produce a decision-ready package with status, severity where applicable, confidence, limitations, actions, residual risk, approval state, and completion criteria.
5. Request independent review for high-impact releases, major architecture shifts, or material residual risk.
